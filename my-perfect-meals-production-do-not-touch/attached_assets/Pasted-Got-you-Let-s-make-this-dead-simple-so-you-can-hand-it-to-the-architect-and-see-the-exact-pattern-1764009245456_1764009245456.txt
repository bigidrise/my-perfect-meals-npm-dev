## 1️⃣ Example TS entry for one feature (Craving Premades)

This is **example structure**, not a full file. Your CanonicalAliasRegistry already exists, so the architect will adapt the field names. But this is the pattern:

```ts
// CanonicalAliasRegistry.ts (EXAMPLE SHAPE)

// Example type – your file will already have something similar
export interface CanonicalFeatureConfig {
  id: string;                // e.g. "CRAVING_PREMADES"
  route: string;             // e.g. "/craving-presets"
  canonicalName: string;     // "Craving Premades"
  uniqueToken: string;       // main distinguishing token
  aliases: string[];         // words/phrases we match on
  samplePhrases: string[];   // for testing + future UI
}

// Somewhere inside your registry object/array:
export const CANONICAL_FEATURES: CanonicalFeatureConfig[] = [
  // ...existing entries...

  {
    id: "CRAVING_PREMADES",
    route: "/craving-presets",
    canonicalName: "Craving Premades",
    uniqueToken: "premades",
    aliases: [
      "craving premades",
      "premades",
      "pre-mades",
      "pre made",
      "pre maid",
      "pro maids",
      "maids",
    ],
    samplePhrases: [
      "Open Craving Premades",
      "Show me premades",
      "Show premade cravings",
      "Take me to premades",
    ],
  },

  // ...more features below...
];
```

If your file uses different fields (like `keywords` instead of `aliases`, or `slug` instead of `route`), the architect will just map them:

* `uniqueToken` → whatever field they use to store the “one safe distinguishing token”.
* `aliases` → whatever array they use for matching (`keywords`, `triggers`, etc.).
* `samplePhrases` → optional; they can keep it just for docs/tests.

The key idea: **one config per feature**, with `uniqueToken` + `aliases`.

### 1. Use this example pattern

For each feature, we want a config with:

* `id` – existing canonical ID (e.g. `CRAVING_PREMADES`, `WEEKLY_MEAL_BUILDER`, etc.)
* `route` – **must match** the existing route in `Router.tsx`
* `canonicalName` – human-readable name
* `uniqueToken` – one distinguishing word for this feature that no other feature should use
* `aliases` – array of tokens/phrases we will match on (case-insensitive)
* `samplePhrases` – (optional) example phrases for tests / future UI
